C++中delete和delete[]的深层区别

发表于 2021-05-26

今天又看到群里有人讨论 C++ 中 delete 和 delete[] 的区别，表层原因大家都了解，因为教科书上说得很明白：new 和 delete 需要配对使用， new[] 和 delete[] 需要配对使用。

但若问起在什么情况下 new[] 可以使用 delete 释放资源而不会有任何问题，能讲清楚这点的人就很少了。因为这涉及到 new、delete、new[]、delete[] 内部实现机制的理解。

简单来说， delete 需要调用一个元素的析构函数，而 delete[] 需要调用数组中若干个元素的析构函数，这里就牵涉到一个问题，就是 delete[] 如何知道数组中元素的数量？我们知道 delete[] 中的 [] 并不传入数值，所以这个数量不是 delete[] 提供过来的，而是 new[] 提供过来的， new[] 的时候需要将这个数量存储起来，以供在 delete[] 的时候能获取到对象的数量。而 new[] 的做法，就是将数量放入申请的动态内存的首地址对应的空间，并且不是返回内存的首地址，而是返回首地址（存放的是数量）之后的下一个内存地址（存放的是第一个元素的数据）。

但是并不是所有的对象类型都需要调用析构函数，比如 int 等基本数据类型就不存在调用析构函数的情况，那么 delete[] 根本无需逐个元素调用析构函数，也就没必要在 new[] 的时候存入元素数量。

还有一种情况，就是并非基本类型而是 struct/class，而且没有显式定义析构函数，这种情况并不会为析构函数产生任何代码，所以就和没有析构函数一样，是不会在 delete[] 的时候逐个调用析构函数，也就如上述的 int 等基本类型的情况一样，不会在 new[] 的时候存入数量。

一眼就能看出，对于 int *ptr = new int[5] 和 A *ptr = new A[5] 其实使用 delete 或者 delete[] 不会有任何区别，因为并不涉及对析构函数的调用，但是对于 B *ptr = new B[5] 却一定要使用 delete[] ，因为传过来的并不是真正的动态内存首地址， delete[] 的内部处理就变成从这个地址之前的地址空间中获取数量并逐个调用元素的析构函数，然后用真正的动态内存首地址释放申请的动态内存。但若使用 delete 就只会调用第一个元素的析构函数，并且将第一个元素的地址作为动态内存的首地址空间去释放，而释放错误的内存地址（非申请时候动态内存的首地址）将发生严重错误，如在 visual studio 中将直接触发程序异常并崩溃。
